# SKILL: Harness Diagnostics (하네스 자가진단)
# 이 스킬은 에이전트가 현재 프로젝트의 하네스 품질을 진단할 때 사용한다.
# 12원칙, 84항목 체크리스트로 정량 점수를 산출한다.

---

## 실행 방법

다음 중 하나를 선택:
- **Setup 모드**: 새 프로젝트에 하네스를 처음 구축할 때
- **Audit 모드**: 현재 프로젝트 종합 평가
- **Maintenance 모드**: 시간이 지난 후 drift 점검

---

## 12가지 평가 원칙

| ID | 원칙 | 핵심 질문 |
|----|------|-----------|
| P1 | Agent Entry Point | AGENTS.md가 존재하고 진입점이 명확한가? |
| P2 | Map, Not Manual | 문서가 목차 역할을 하는가? (200줄 이하) |
| P3 | Invariant Enforcement | lint/test/type-check가 자동 실행·차단되는가? |
| P4 | Convention Over Config | 명시적 컨벤션 문서가 있는가? |
| P5 | Progressive Disclosure | 필요한 정보를 필요한 시점에 접근할 수 있는가? |
| P6 | Layered Architecture | 의존성 방향이 명확하고 강제되는가? |
| P7 | Garbage Collection | 낡은 코드/문서가 정기적으로 정리되는가? |
| P8 | Observability | 에이전트가 작업 결과를 스스로 검증할 수 있는가? |
| P9 | Knowledge in Repo | 지식이 레포에 있는가? (슬랙/머릿속이 아닌) |
| P10 | Reproducibility | 같은 입력에 같은 결과가 보장되는가? |
| P11 | Modularity | 변경 영향 범위가 예측 가능한가? |
| P12 | Self-Documentation | 코드 자체가 의도를 설명하는가? |

---

## 체크리스트 (Audit 모드)

### 카테고리 1: Agent Entry Point (P1) — 15항목
- [ ] 레포 루트에 `AGENTS.md` 파일이 존재하는가?
- [ ] 프로젝트 한 줄 요약이 있는가?
- [ ] 기술 스택이 명시되어 있는가?
- [ ] 빌드 명령어가 복사-붙여넣기 가능한 형태로 기재되어 있는가?
- [ ] 테스트 실행 명령어가 기재되어 있는가?
- [ ] 디렉토리 구조 설명이 있는가?
- [ ] 코딩 컨벤션 링크/요약이 있는가?
- [ ] 흔한 실수/주의사항(gotchas)이 기재되어 있는가?
- [ ] 환경 변수 설정 안내가 있는가?
- [ ] 관련 문서 링크가 있는가?
- [ ] AGENTS.md가 200줄 이하인가? (목차 역할)
- [ ] 세부 규칙이 별도 파일로 분리되어 있는가?
- [ ] 최근 코드 변경사항을 반영하고 있는가?
- [ ] 작업 루프(5단계)가 명시되어 있는가?
- [ ] 절대 원칙(3가지)이 명시되어 있는가?

### 카테고리 2: 문서 구조 (P2, P5, P9) — 12항목
- [ ] `docs/` 디렉토리가 존재하는가?
- [ ] `docs/` 내부가 카테고리별로 구조화되어 있는가?
- [ ] ADR(Architecture Decision Records)이 존재하는가?
- [ ] `docs/project/PLANS.md`가 작성되어 있는가?
- [ ] `docs/design-docs/`에 아키텍처 원칙이 있는가?
- [ ] `docs/skills/`에 재사용 가능한 스킬이 있는가?
- [ ] README.md가 기본 정보를 포함하는가?
- [ ] 문서 간 상호 참조(cross-reference)가 있는가?
- [ ] 코드 내 주석이 "why"를 설명하는가?
- [ ] CHANGELOG 또는 릴리스 노트가 있는가?
- [ ] 환경 변수 템플릿(`.env.template`)이 있는가?
- [ ] 기술 결정이 레포에 기록되는가? (ADR, PR)

### 카테고리 3: Invariant 강제 (P3, P4, P10) — 15항목
- [ ] Linter 설정이 존재하는가?
- [ ] Formatter 설정이 존재하는가?
- [ ] CI/CD 파이프라인이 존재하는가?
- [ ] CI에서 테스트가 실행되는가?
- [ ] CI에서 lint가 실행되는가?
- [ ] Pre-commit hook이 설정되어 있는가?
- [ ] 커밋 메시지 형식이 강제되는가? (commitlint)
- [ ] 금지 패턴(console.log 등)이 커밋 시 차단되는가?
- [ ] Branch protection이 설정되어 있는가?
- [ ] 보안 취약점 스캔이 자동 실행되는가?
- [ ] 의존성 자동 업데이트가 설정되어 있는가? (Dependabot)
- [ ] 테스트 커버리지 기준이 있는가?
- [ ] docker-compose로 로컬 환경 재현이 가능한가?
- [ ] 빌드가 항상 동일한 결과를 내는가?
- [ ] 시크릿이 코드에 없는가? (`.env` gitignore 확인)

### 카테고리 4: 아키텍처 (P6, P11) — 12항목
- [ ] 레이어 분리가 되어 있는가?
- [ ] 의존성 방향이 일관적인가?
- [ ] 순환 의존성이 없는가?
- [ ] 도메인 간 직접 의존이 없는가?
- [ ] 외부 API 호출이 추상화 레이어를 통하는가?
- [ ] 에러 처리 패턴이 프로젝트 전반에서 일관적인가?
- [ ] 설정과 코드가 분리되어 있는가?
- [ ] 테스트 구조가 소스 구조와 대칭적인가?
- [ ] 공통 유틸리티가 중앙화되어 있는가?
- [ ] 모듈 경계가 명확한가?
- [ ] 패키지 구조가 도메인 중심인가?
- [ ] API contract가 명시적으로 정의되어 있는가?

### 카테고리 5: 운영/유지보수 (P7, P8) — 10항목
- [ ] 에이전트 실행 로그가 저장되는가?
- [ ] 검증 실패 로그가 저장되는가?
- [ ] Slack 알림이 설정되어 있는가?
- [ ] 낡은 태스크가 archive로 이동되는가?
- [ ] drift 스캔이 주기적으로 실행되는가?
- [ ] 의존성이 주기적으로 업데이트되는가?
- [ ] Dead code가 주기적으로 제거되는가?
- [ ] 에러 모니터링 도구가 연동되어 있는가?
- [ ] Health check 엔드포인트가 있는가?
- [ ] Flaky test가 관리되고 있는가?

```
총점 = (카테고리1/15 + 카테고리2/12 + 카테고리3/15 + 카테고리4/12 + 카테고리5/10) / 5 × 100
```

## 성숙도 레벨

| 점수 | 레벨 | 설명 |
|------|------|------|
| 0-19 | L1: None | 에이전트 협업 미고려 |
| 20-39 | L2: Basic | 최소한의 문서만 존재 |
| 40-59 | L3: Structured | 체계적 구조, 부분 자동화 |
| 60-79 | L4: Optimized | 높은 자동화, 낮은 drift |
| 80-100 | L5: Autonomous | 에이전트가 독립 기여 가능 |
